# PR 26 partial fulfillment technical debt

Context: PR #26 added partial fulfillment for authorized requisitions through `POST /api/v1/requisitions/{id}/fulfill/` and was hardened across review commits `6b418ff` and `200777a` on branch `feat/atendimento-parcial`. The PR delivered the unified `atender_requisicao()` service entry point, per-item partial fulfillment, `LIBERACAO_RESERVA_ATENDIMENTO`, deterministic stock locks, API/docs/test coverage for all-zero rejection, and `400 validation_error` for payload coherence problems.

Tracked debt:
- Idempotency is not implemented for client retries after a network failure. The current behavior is concurrency-safe because the first successful fulfillment changes requisition status under lock and a second request returns `409 domain_conflict`, but the API does not replay the original successful response. Revisit with an idempotency-key design if external clients or unreliable networks make retry semantics important.
- The fulfillment service locks `Requisicao`, then authorized `ItemRequisicao` rows ordered by `material_id, id`, then related `EstoqueMaterial` rows in deterministic `material_id` order before calling stock writers with `estoque_travado`. Future stock writers such as returns, reversals, exceptional exits, or cancellation release must reuse a canonical lock order and add PostgreSQL concurrency coverage rather than introducing a parallel lock sequence. Stock writers that receive `estoque_travado` now require an active transaction.
- `atender_requisicao()` is now the single service entry point used by the DRF `fulfill` action, but `atender_requisicao_completa()` and `atender_requisicao_com_itens()` still exist for compatibility/tests. A future refactor may consolidate their shared validation/lock code if another fulfillment variant is introduced, but avoid broad refactor until there is concrete duplication pressure.
- PR #26 review decisions: duplicated, unknown, or omitted `item_id` in fulfillment payload is input/payload incoherence and should be `ValidationError` / HTTP `400 validation_error`; all authorized items delivered as zero is a domain-rule conflict and should remain `409 domain_conflict` with no state, stock, or movement side effects.
